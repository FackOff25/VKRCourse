\subsection{Структуры данных}

Для реализации программы требуется реализовать следующие структуры данных:

\begin{itemize}
    \item Граф, включая:
    \begin{itemize}
        \item Ключ вершины.
        \item Вершина графа.
        \item Ребро графа. 
    \end{itemize}
    \item Хранилище.
\end{itemize}

Выбор структур данных является одним из наиболее важных этапов проектирования системы. Именно используемые структуры определяют удобство реализации алгоритмов распределения, скорость доступа к данным, объём занимаемой памяти и сложность взаимодействия между отдельными компонентами распределённой среды.

Так как основной задачей разрабатываемой системы является исследование алгоритмов распределения графовых данных между несколькими хранилищами, при проектировании структур данных приоритет отдавался не максимальной производительности, а простоте реализации, расширяемости и возможности быстрого прототипирования различных алгоритмов. Такой подход позволяет сосредоточиться непосредственно на исследуемых методах распределения и оптимизации, минимизируя влияние второстепенных факторов.

При этом все структуры данных разрабатывались с учётом возможности дальнейшего расширения. В частности, в реализации отсутствует жёсткая привязка к конкретному типу ключей вершин или конкретному набору параметров, что позволяет использовать разработанную систему для моделирования широкого спектра графовых данных.

Использование отдельной структуры для представления ключа вершины позволяет абстрагироваться от конкретного типа идентификатора. В реальных графовых системах в качестве ключей могут использоваться целые числа, строки, UUID, составные ключи и другие типы данных. Поддержка произвольного типа ключа делает систему более универсальной и позволяет применять её для различных предметных областей без изменения внутренней логики алгоритмов.

Кроме того, выделение ключа в отдельную структуру упрощает реализацию операций сравнения, поиска и хранения вершин в индексированных контейнерах. Это особенно важно для алгоритмов распределения и маршрутизации запросов, которые постоянно выполняют поиск вершин по их идентификаторам.

Вершина является центральным объектом графовой модели и представляет собой минимальную единицу хранения информации. Каждая вершина объединяет идентификатор объекта, произвольный набор его характеристик и информацию о связях с другими вершинами графа.

Использование бинарного массива для хранения параметров обусловлено необходимостью поддержки произвольной структуры данных. В рамках разрабатываемой системы содержимое параметров не участвует в алгоритмах распределения и поиска пути напрямую. Вместо этого параметры рассматриваются как пользовательские данные, которые могут содержать любую информацию, необходимую конкретному сценарию использования системы.

Такой подход позволяет отделить механизмы хранения и распределения данных от их предметной интерпретации. В результате одна и та же реализация может использоваться для моделирования социальных графов, транспортных сетей, графов зависимостей программных компонентов или любых других графовых структур.

Особенностью представленной структуры является хранение исходящих рёбер непосредственно внутри вершины. Подобный подход упрощает получение соседей вершины и позволяет выполнять локальные операции обхода графа без обращения к дополнительным структурам. Несмотря на то, что в крупных промышленных графовых системах обычно применяются более сложные схемы хранения, для целей исследования и тестирования алгоритмов выбранное решение является достаточно удобным и наглядным.

Ребро используется для представления отношения между двумя вершинами графа. Помимо стандартной информации о соединяемых вершинах структура содержит дополнительные данные, необходимые для реализации алгоритмов оптимизации распределения.

Особое значение имеет поле запроса веса ребра. В отличие от классического веса ребра, который обычно характеризует расстояние, стоимость перехода или пропускную способность канала, данный параметр отражает интенсивность использования связи в процессе обработки запросов. Первоначально все рёбра имеют одинаковый вес, однако в ходе работы системы этот параметр изменяется в зависимости от частоты использования ребра при поиске путей.

Подобный механизм позволяет постепенно формировать модель реальной нагрузки на граф. Рёбра, которые часто участвуют в выполнении запросов, становятся более значимыми для системы. В дальнейшем эта информация используется алгоритмом структурной оптимизации для уменьшения числа дорогостоящих межшардовых переходов.

Дополнительным преимуществом выбранной структуры является поддержка как ориентированных, так и неориентированных графов. Это расширяет область применения разработанной системы и позволяет использовать её для моделирования различных классов задач.

Хранилище должно содержать информацию обо всех рёбрах внутри хранилища, а также рёбра, "исходящие" из него, для получения граничных вершин. Для этого была разработана древовидная структура данных состоящая из вложенных индексированных списков, названная "хранилище ветвей" и представленная на рисунке \ref{project-fig:external_edges}.

Необходимость хранения информации о внешних связях обусловлена особенностями работы алгоритмов оптимизации распределения. Для определения качества текущего разбиения требуется быстро получать сведения о рёбрах, пересекающих границы хранилищ. Именно такие рёбра формируют разрез графа и оказывают наибольшее влияние на количество межшардовых переходов.

Предложенная структура позволяет организовать данные таким образом, чтобы для каждого удалённого хранилища можно было быстро определить набор вершин и рёбер, связывающих текущее хранилище с этим удалённым сегментом графа. Это существенно упрощает реализацию алгоритма Кернигана--Лина и позволяет эффективно определять граничные вершины при выполнении оптимизации.

Кроме того, наличие отдельной структуры хранения внешних связей позволяет избежать полного обхода графа при вычислении характеристик разреза. Это особенно важно при увеличении числа вершин и рёбер, когда стоимость полного анализа структуры графа начинает существенно возрастать.

Хранилище представляет собой базовую единицу распределения данных в системе. Каждое хранилище отвечает за хранение некоторого подмножества вершин графа и всех внутренних рёбер между ними.

Поле идентификатора используется для однозначного определения хранилища внутри распределённой среды. Наличие уникального идентификатора позволяет организовывать обмен сообщениями между компонентами системы и поддерживать информацию о расположении вершин.

Основной объём данных хранится в структуре \texttt{nodes}, содержащей все вершины, закреплённые за данным хранилищем. Использование индексированного контейнера обеспечивает быстрый доступ к вершинам по их ключам, что является критически важным для выполнения запросов поиска пути и операций перераспределения данных.

Поле \texttt{external\_edges} используется для хранения информации о связях с другими хранилищами и играет ключевую роль при выполнении структурной оптимизации. Именно на основе этих данных определяются граничные вершины, оценивается качество разбиения и принимаются решения о переносе вершин между хранилищами.

Указатель на шину обеспечивает взаимодействие хранилища с остальными компонентами системы. Через шину выполняются рассылка уведомлений о перемещении вершин, запрос информации о расположении данных и обмен служебной информацией между элементами распределённой среды.

Таким образом, предложенный набор структур данных обеспечивает поддержку всех операций, необходимых для реализации потокового распределения, структурной оптимизации, учёта нагрузки и выполнения запросов в распределённой графовой системе. При этом выбранные решения остаются достаточно простыми для экспериментального исследования и последующей модификации при необходимости расширения функциональности системы.
